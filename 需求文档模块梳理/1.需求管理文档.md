## 需求管理功能说明

### 一、模块概述
在现有“埋点管理”目录下新增“需求管理”子模块。用于管理需求的完整生命周期，包括需求创建、状态流转、权限控制、列表查询与筛选。

### 二、列表页面

#### 2.1 页面路径
`/requirement-manage`（示例）

#### 2.2 表格列定义
| 列名 | 数据字段 | 说明 |
|------|----------|------|
| 需求ID | `requirementId` | 系统生成UUID，字符串 |
| 需求标题 | `title` | 蓝色字体，可点击打开详情Tab |
| 需求状态 | `status` | 见状态枚举 |
| 优先级 | `priority` | P0(高)、P1(中)、P2(低) |
| 需求提出人 | `proposerName` | 用户昵称 |
| 所属业务线 | `businessLine` | 从字典接口获取（见2.4） |
| 所属部门 | `department` | 字符串，由用户信息关联 |
| 创建时间 | `createTime` | datetime，格式 `YYYY-MM-DD HH:mm:ss` |
| 更新时间 | `updateTime` | datetime，格式 `YYYY-MM-DD HH:mm:ss` |
| 操作 | `actions` | 根据状态和角色展示不同按钮（见五） |

#### 2.3 筛选条件（位于表格上方）
- **可筛选字段**：需求标题（文本模糊）、需求状态（下拉多选）、优先级（下拉）、需求提出人（文本）、所属业务线（下拉）、所属部门（文本）
- **重置按钮**：一键清空所有筛选条件，恢复初始列表数据
- **列表分页**：列表需分页，默认以创建时间排序，可对优先级、创建时间、更新时间进行排序。

#### 2.4 字典数据来源
- 所属业务线和负责开发团队下拉选项：调用 `/dict-param/ids-list` 接口，支持id数组请求
- 前端需过滤：`status === 0` 的项

#### 2.5 列表数据刷新策略
- 初始加载、筛选后、提交新增/编辑/状态变更后，均需刷新列表。

### 三、新增需求页面

#### 3.1 表单字段

| 字段 | 组件类型 | 是否必填 | 数据源/备注 |
|------|----------|----------|--------------|
| 需求标题 | Input | **是** | 文本，最大长度200 |
| 所属业务线 | Select | **是** | 下拉，数据来自2.4字典 |
| 优先级 | Select | **是** | 选项：P0高 / P1中 / P2低 |
| 期望上线时间 | DatePicker | **是** | 日期，格式 YYYY-MM-DD |
| 负责开发团队 | Select | **是** | 下拉，数据来自2.4字典 |
| 需求描述 | Textarea | 否 | 纯文本，不限长度，提示请输入需求的详细背景、目标和业务逻辑（非必填） |

> **说明**：除需求描述，其他字段均为**必填**

#### 3.2 底部按钮
- **取消**：清空表单内容并返回列表页
- **提交**：校验必填项 → 调用创建接口 → 成功后关闭页面并刷新列表

#### 3.3 提交逻辑
- 生成唯一 `requirementId`（uuid v4）
- 记录当前时间作为 `createTime`
- 记录当前需求提出人，昵称和id
- 需求状态初始化为 `待审核`（status = "PENDING_REVIEW"）
- 将表单数据与自动生成字段一并存入后端

### 四、需求详情（Tab页）

**触发方式**：在列表页点击 **需求标题（蓝色字体）**

**展示内容**：只读模式，包含以下字段：

- 需求ID（uuid）
- 需求状态
- 需求标题
- 需求创建时间
- 所属业务线
- 所属部门
- 需求提出人
- 优先级
- 期望上线时间
- 负责开发团队
- 需求描述
- 操作日志列表（见7.4）

**底部按钮**：`关闭`（关闭Tab返回列表）

### 五、需求状态枚举与流转规则

#### 5.1 状态列表（代码 & 显示名）
| 代码 | 显示名 |
|------|--------|
| `PENDING_REVIEW` | 待审核 |
| `SCHEDULING` | 排期中 |
| `DEVELOPING` | 开发中 |
| `TESTING` | 测试中 |
| `ONLINE` | 已上线 |
| `OFFLINE` | 已下线 |
| `REJECTED` | 审核不通过 |

#### 5.2 状态流转规则（角色权限详见六）

| 当前状态 | 允许的目标状态 | 允许的操作角色 |
|----------|----------------|----------------|
| 待审核（新增后） | 排期中、审核不通过 | 仅管理员 |
| 排期中 | 开发中、待审核 | 开发中：管理员或开发人员；待审核：仅管理员 |
| 开发中 | 测试中、排期中 | 管理员或开发人员 |
| 测试中 | 已上线、开发中 | 管理员或开发人员 |
| 已上线 | 已下线、测试中 | 已下线：仅管理员；测试中：管理员或开发人员 |
| 已下线 | 已上线 | 仅管理员 |
| 审核不通过 | 无直接变更，仅可“编辑重新提交” | 管理员 或 该需求的提出人 |

> **注意**：
>
> - “向后变更”通常表示流程推进，但规则中允许部分回退（如排期中→待审核、开发中→排期中等），请按表格实现。

### 六、操作列按钮逻辑（列表页操作列）

根据当前登录用户的**角色**和需求的**状态**，展示不同的按钮（可能为“变更状态”下拉/一组按钮，或“编辑”按钮）。

变更状态下拉一组按钮，点击后需二次弹窗确认，例如：请确认是否将此{需求标题}需求变更为***状态？点击确认即提交并刷新列表。点击审核不通过时需要输入原因，记入到操作日志里的操作意见里。

#### 6.1 角色定义
- **管理员**：拥有所有状态变更权限（除个别限制）以及对“审核不通过”需求的编辑权。
- **开发人员**：拥有部分状态向前或回退权限（如上表所示）。
- **需求提出人**：仅对状态为“审核不通过”且是自己提出的需求，显示“编辑”按钮。

#### 6.2 各状态下显示的按钮（合并变更操作为单个“变更状态”下拉或分按钮）

| 需求状态 | 管理员 | 开发人员 | 需求提出人 |
|----------|--------|----------|-------------|
| 待审核 | 变更状态按钮下拉有：“排期中”、“审核不通过” | 无按钮 | 无按钮 |
| 排期中 | 变更状态按钮下拉有：“待审核”、“开发中” | 按钮：“变更为开发中” | 无按钮 |
| 开发中 | 变更状态按钮下拉有：“测试中”、“排期中” | 按钮：“变更为测试中”、“变更为排期中” | 无按钮 |
| 测试中 | 变更状态按钮下拉有：“已上线”、“开发中” | 按钮：“变更为已上线”、“变更为开发中” | 无按钮 |
| 已上线 | 变更状态按钮下拉有：“已下线”、“测试中” | 按钮：“变更为测试中” | 无按钮 |
| 已下线 | 变更状态按钮下拉有：“已上线” | 无按钮 | 无按钮 |
| 审核不通过 | 显示按钮：“编辑” | 无按钮 | 按钮：“编辑” |

> **说明**：
> - 点击“编辑”按钮 → 打开编辑页面（见第七节）。
> - 点击变更状态按钮 → 弹出确认框 → 调用状态变更接口 → 刷新列表。

### 七、审核不通过后的编辑重新提交

#### 7.1 触发条件
- 需求状态为 `REJECTED`
- 当前登录用户为 **管理员** 或 **该需求的提出人**

#### 7.2 编辑页面
- 可修改字段：同新增页面所有字段（需求标题、业务线、优先级、期望上线时间、负责开发团队、需求描述）
- **特殊校验**：只有至少一个字段发生实质性修改时，才允许提交（前端比较原值与新值）
- **其他逻辑**：原有字段值预填

#### 7.3 重新提交逻辑
- 记录 **更新时间**（即提交时刻）
- 状态直接变为 `待审核`
- 提交成功后，列表刷新

#### 7.4 操作日志记录

所有提交，变更状态，编辑操作需记录操作日志，查看页面时页面底部展示。列表记录操作人，操作时间，操作类型，变更前后状态值， 操作意见。

### 八、补充说明
- 需求ID生成使用 uuid v4，建议后端生成以保证唯一性。
- 所有时间字段使用服务器时间。
- 操作列按钮应根据后端返回的当前用户角色和状态动态渲染（由后端接口告知可用操作）。
- 业务线和开发团队的列表获取，字典ID是不会变化的，前端直接请求ids-list接口，接收参数id数组请求，具体可看代码。
- 需求提出人即为业务人员角色，现有系统已有三种角色，管理员、开发人员、业务人员。
### 九、上传图片功能补充（2026-04-27）

#### 9.1 字段补充
- 需求新增截图字段：screenshotFileId（非必填）。
- 字段含义：关联统一文件表中的图片文件ID。

#### 9.2 上传规则
- 支持格式：jpg、png。
- 文件大小：最大 2MB。
- 不符合格式或大小限制时，前后端均需拦截并给出提示。

#### 9.3 页面交互
- 在“新增需求”表单中增加“需求截图”上传项。
- 在“需求详情”中可查看已上传截图。
- 在“审核不通过后的编辑重提”页中可修改截图。
- 上传组件需支持：上传、查看、移除、动态格式与大小配置。

#### 9.4 后端存储与安全预览
- 图片统一存储在 `track_file_asset` 表。
- 需求表 `track_requirement.screenshot_file_id` 保存关联ID。
- 图片预览通过鉴权接口返回二进制（blob）实现，不使用免登录静态地址。

#### 9.5 接口补充
- POST /api/file/upload-image：上传图片。
- GET /api/file/preview/{fileId}：预览图片（需登录鉴权）。
- POST /api/requirement/add：新增需求时可携带 screenshotFileId。
- POST /api/requirement/resubmit：编辑重提时可携带 screenshotFileId。
- GET /api/requirement/detail：返回 screenshotFileId。
