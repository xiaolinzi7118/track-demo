## 属性管理功能说明

### 一、模块概述

在现有"埋点管理"目录下新增"属性管理"子模块。用于统一维护埋点上报所需的属性，供事件管理引用。属性分为三种类型：用户属性、系统属性和个性化属性，SDK 上报时根据属性类型按不同规则取值。

路由路径：`/attribute-manage`

### 二、属性类型与 SDK 取值规则

| 属性类型 | attributeType | SDK 取值逻辑 | 示例 |
|---|---|---|---|
| 用户属性 | `user` | `window.userInfo.{attributeField}` | attributeField=`name` → `window.userInfo.name` |
| 系统属性 | `system` | `window.system.{attributeField}` | attributeField=`os` → `window.system.os` |
| 个性化属性 | `custom` | 按 `sourceType` 取值（5种来源类型） | sourceType=`global_object`，sourceValue=`userInfo.token` → `window.userInfo.token` |

### 三、列表页面

#### 3.1 页面路径

`/attribute-manage`

#### 3.2 表格列定义

| 列名 | 数据字段 | 说明 |
|---|---|---|
| 属性名称 | `attributeName` | 属性的显示名称 |
| 属性字段 | `attributeField` | 上报时的字段 key，如 `name`、`os`、`productId` |
| 属性类型 | `attributeType` | Tag 展示：用户属性 / 系统属性 / 个性化属性 |
| SDK 取值路径 | - | 只读展示，自动推导：用户属性→`window.userInfo.{field}`，系统属性→`window.system.{field}`，个性化属性→按来源类型展示简要说明 |
| 来源类型 | `sourceType` | 仅个性化属性展示，显示中文名（见 4.3） |
| 状态 | `status` | `0=生效`、`1=已删除`；Tag 展示 |
| 创建时间 | `createTime` | `YYYY-MM-DD HH:mm:ss` |
| 操作 | - | 编辑、删除 |

#### 3.3 筛选条件（位于表格上方）

- **属性名称**：文本模糊搜索
- **属性字段**：文本模糊搜索
- **属性类型**：下拉单选（用户属性 / 系统属性 / 个性化属性）
- **重置按钮**：清空所有筛选条件，恢复初始列表
- **列表分页**：默认按创建时间降序，支持分页

#### 3.4 列表数据刷新策略

初始加载、筛选后、提交新增/编辑/删除后，均需刷新列表。

### 四、新增/编辑属性（弹窗）

#### 4.1 通用表单字段

| 字段 | 组件类型 | 是否必填 | 说明 |
|---|---|---|---|
| 属性名称 | Input | **是** | 最大 100 字符；生效数据内唯一 |
| 属性字段 | Input | **是** | 最大 100 字符；如 `name`、`os`、`productId` |
| 属性类型 | Select | **是** | 用户属性 / 系统属性 / 个性化属性 |

#### 4.2 个性化属性额外字段（`attributeType=custom` 时展示）

| 字段 | 组件类型 | 是否必填 | 说明 |
|---|---|---|---|
| 来源类型 | Select | **是** | 见 4.3 |
| 接口选择 | Select | 条件必填 | 来源类型为"接口数据"时必填，下拉展示接口管理中的接口列表（接口名称 + 路径） |
| 变量路径/值 | Input | 条件必填 | 来源类型非"节点内容"时需填写；placeholder 见 4.3 |
| 默认值 | Input | 否 | 取值失败时的兜底值 |

> **说明**：属性类型切换时，个性化属性的额外字段清空。

#### 4.3 个性化属性来源类型枚举

| sourceType | 显示名 | 变量路径/值 placeholder | 使用限制 |
|---|---|---|---|
| `node_content` | 节点内容 | 无需填写（自动获取点击节点文本） | 仅 `click` 类型事件可用 |
| `api_data` | 接口数据 | 如：`data.productName` | 需选择接口；仅 `click` 类型事件可用 |
| `global_object` | 全局对象 | 如：`userInfo.id` | 所有事件类型可用 |
| `local_cache` | 本地缓存 | 如：`token`（localStorage key） | 所有事件类型可用 |
| `static_value` | 静态值 | 如：`male` | 所有事件类型可用 |

#### 4.4 底部按钮

- **取消**：关闭弹窗，不保存
- **确定**：校验必填项 → 调用新增/更新接口 → 成功后关闭弹窗并刷新列表

### 五、删除逻辑

- 点击"删除"弹出二次确认框：`确认删除属性「{属性名称}」？`
- 后端校验：若该属性已被任意事件引用（即 `track_config.params` 中存在对该 `attributeId` 的引用），则拒绝删除并提示"该属性已被事件引用，无法删除"
- 删除为软删除，置 `status=1`

### 六、数据模型

#### 6.1 新建表：`track_attribute`

```sql
CREATE TABLE track_attribute (
  id              BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键',
  attribute_id    VARCHAR(32) NOT NULL COMMENT '业务ID，格式：ATTR+yyyyMMdd+8位序号',
  attribute_name  VARCHAR(100) NOT NULL COMMENT '属性名称',
  attribute_field VARCHAR(100) NOT NULL COMMENT '属性字段，如 name / os / productId',
  attribute_type  VARCHAR(20) NOT NULL COMMENT '属性类型：user=用户属性, system=系统属性, custom=个性化属性',
  source_type     VARCHAR(50) DEFAULT NULL COMMENT '来源类型（custom 时有效）',
  source_value    VARCHAR(500) DEFAULT NULL COMMENT '变量路径或静态值',
  interface_id    BIGINT DEFAULT NULL COMMENT '接口ID（api_data 时有效）',
  interface_path  VARCHAR(500) DEFAULT NULL COMMENT '接口路径，提交时由后端根据 interface_id 自动补充',
  default_value   VARCHAR(255) DEFAULT NULL COMMENT '取值失败时的默认值',
  status          TINYINT NOT NULL DEFAULT 0 COMMENT '0=生效, 1=已删除',
  create_by       VARCHAR(64) DEFAULT NULL,
  update_by       VARCHAR(64) DEFAULT NULL,
  create_time     DATETIME(3) DEFAULT NULL,
  update_time     DATETIME(3) DEFAULT NULL,
  PRIMARY KEY (id),
  UNIQUE KEY uk_attribute_id (attribute_id)
) COMMENT='属性管理表';
```

**业务 ID 生成规则**：`ATTR + yyyyMMdd + 8位序号`，复用 `dict_id_sequence` 表生成（新增序列类型 `ATTR`）。

#### 6.2 TrackAttribute 数据结构

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | number | 主键 |
| `attributeId` | string | 业务 ID，如 `ATTR2026042600000001` |
| `attributeName` | string | 属性名称 |
| `attributeField` | string | 属性字段 |
| `attributeType` | string | `user` / `system` / `custom` |
| `sourceType` | string \| null | 来源类型枚举（`custom` 时有效） |
| `sourceValue` | string \| null | 变量路径或静态值 |
| `interfaceId` | number \| null | 接口 ID（`api_data` 时有效） |
| `interfacePath` | string \| null | 接口路径（后端自动填充） |
| `defaultValue` | string \| null | 默认值 |
| `status` | number | `0=生效`，`1=已删除` |
| `createBy` | string | 创建人 |
| `updateBy` | string | 更新人 |
| `createTime` | datetime | 创建时间 |
| `updateTime` | datetime | 更新时间 |

- 方法：`POST`

**请求体**

| 字段 | 类型 | 必填 | 说明 |

### 八、业务规则说明

1. 属性名称唯一性：同一 `attributeType` 下，`attributeName` 在生效数据（`status=0`）内唯一。
2. 已被事件引用的属性不可删除，后端在删除前校验 `track_config.params` 中的 `attributeId` 引用。
3. 个性化属性的 `sourceType=node_content` 或 `api_data` 仅适用于 `click` 类型事件；在事件管理的属性选择器中，若当前事件类型为 `page_view`，这两类属性不可选（置灰提示）。
4. `interfacePath` 字段由后端根据 `interfaceId` 自动查询并填充，前端无需传入。
5. 属性 ID 生成规则：`ATTR + yyyyMMdd + 8位序号`。
