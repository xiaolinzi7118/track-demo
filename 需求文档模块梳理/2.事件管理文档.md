## 事件管理功能说明

### 一、模块概述

原"埋点配置"模块更名为"**事件管理**"，路由路径从 `/track-config` 改为 `/event-manage`，菜单名称同步更新为"事件管理"。

本次改造内容：

1. **新增关联需求**：创建/编辑事件时，必须关联一条状态为"排期中"或"开发中"的需求（1:1 关联）。
2. **移除独立 URL 正则配置**：原"生效页面 URL 正则"输入项不再作为独立字段暴露，`url_pattern` 字段保留以兼容历史数据。
3. **参数配置改为关联属性**：原内联编辑的参数配置表格，替换为从属性管理中选择属性的方式。用户属性和系统属性直接引用已创建的属性条目；个性化属性在引用的同时可在行内维护其来源类型配置（继承现有5种来源类型编辑逻辑）。每条已选属性均可添加属性描述。

### 二、列表页面

#### 2.1 页面路径

`/event-manage`

#### 2.2 表格列定义

| 列名 | 数据字段 | 说明 |
|---|---|---|
| 事件名称 | `eventName` | - |
| 事件编码 | `eventCode` | - |
| 事件类型 | `eventType` | Tag 展示：`page_view`=页面曝光 / `click`=点击交互 |
| 关联需求 | `requirementTitle` | 展示需求标题文字 + 需求状态 Badge |
| 状态 | `status` | Tag 展示：`1=启用`（success） / `0=禁用`（danger） |
| 创建时间 | `createTime` | `YYYY-MM-DD HH:mm:ss` |
| 操作 | - | 编辑、删除 |

#### 2.3 筛选条件（位于表格上方）

- **事件类型**：下拉单选（页面曝光 / 点击交互）
- **关键词**：按事件名称、事件编码模糊搜索
- **重置按钮**：清空所有筛选条件，恢复初始列表
- **列表分页**：默认按创建时间降序

#### 2.4 列表数据刷新策略

初始加载、筛选后、提交新增/编辑/删除后，均需刷新列表。

### 三、新增/编辑事件（弹窗）

#### 3.1 表单字段

| 字段 | 组件类型 | 是否必填 | 说明 |
|---|---|---|---|
| 事件名称 | Input | **是** | - |
| 事件编码 | Input | **是** | - |
| 事件类型 | Select | **是** | 页面曝光（`page_view`） / 点击交互（`click`） |
| 状态 | RadioGroup | **是** | 启用 / 禁用 |
| 关联需求 | Select（可搜索） | **是** | 仅展示状态为"排期中"或"开发中"的需求；选项显示格式：`需求标题 [状态]` |
| 描述 | Textarea | 否 | - |
| 关联属性 | 属性选择表格 | 否 | 见 3.2 |

#### 3.2 关联属性交互

**添加属性：**

- 点击"添加属性"按钮，弹出属性选择弹窗
- 弹窗中展示全量属性列表（来自 `/api/attribute/all`），支持按属性名称、属性字段搜索
- 当前事件类型为 `page_view` 时，`sourceType` 为 `node_content` 或 `api_data` 的个性化属性置灰，不可选（hover 提示"仅点击交互事件可用"）
- 已添加的属性不可重复选择

**已选属性展示（内嵌表格）：**

| 列名 | 说明 |
|---|---|
| 属性名称 | 只读，来自属性管理 |
| 属性类型 | Tag 展示：用户属性 / 系统属性 / 个性化属性 |
| 属性字段 | 只读，来自属性管理 |
| SDK 取值路径 | 只读，自动推导展示（见属性管理第二节） |
| 属性描述 | 可编辑文本框，用于说明该属性在当前事件中的用途 |
| 来源类型配置 | **仅个性化属性展示此列**，行内展示来源类型下拉 + 变量路径/值输入 + 接口选择（api_data 时）+ 默认值输入；编辑逻辑与字段含义同属性管理中的个性化属性配置一致 |
| 操作 | "删除"按钮，移除该行 |

> **说明**：个性化属性在事件中的来源类型配置（`sourceType`、`sourceValue`、`interfaceId`、`defaultValue`）存储在 `track_config.params` 对应条目中，不回写属性管理表，允许同一属性在不同事件中使用不同的来源配置。

#### 3.3 底部按钮

- **取消**：关闭弹窗，不保存
- **确定**：校验必填项 → 调用新增/更新接口 → 成功后关闭弹窗并刷新列表

### 四、删除逻辑

点击"删除"弹出二次确认框：`确认删除事件「{事件名称}」？`，确认后调用删除接口并刷新列表。

### 五、数据模型变更

#### 5.1 `track_config` 表变更

**新增字段：**

| 字段 | 类型 | 说明 |
|---|---|---|
| `requirement_id` | VARCHAR(36) | 关联需求 ID，对应 `track_requirement.requirement_id` |

**字段结构调整：**

- `params`（TEXT）：JSON 数组结构变更，由原来的内联参数配置改为属性引用列表，结构见 5.2。
- `url_pattern`（TEXT）：字段保留（历史兼容），UI 不再暴露独立配置入口。

#### 5.2 `params` 字段新结构

`params` 存储当前事件关联的属性列表，每个元素对应一条已选属性：

```json
[
  {
    "attributeId": "ATTR2026042600000001",
    "attributeType": "user",
    "attributeName": "用户昵称",
    "attributeField": "nickname",
    "description": "当前登录用户的昵称",
    "sourceType": null,
    "sourceValue": null,
    "interfaceId": null,
    "interfacePath": null,
    "defaultValue": null
  },
  {
    "attributeId": "ATTR2026042600000002",
    "attributeType": "system",
    "attributeName": "操作系统",
    "attributeField": "os",
    "description": "用户设备操作系统",
    "sourceType": null,
    "sourceValue": null,
    "interfaceId": null,
    "interfacePath": null,
    "defaultValue": null
  },
  {
    "attributeId": "ATTR2026042600000003",
    "attributeType": "custom",
    "attributeName": "商品ID",
    "attributeField": "productId",
    "description": "当前浏览的商品ID",
    "sourceType": "api_data",
    "sourceValue": "data.productId",
    "interfaceId": 1,
    "interfacePath": "/api/product/detail",
    "defaultValue": ""
  }
]
```

**字段说明：**

| 字段 | 类型 | 说明 |
|---|---|---|
| `attributeId` | string | 对应 `track_attribute.attribute_id` |
| `attributeType` | string | `user` / `system` / `custom`，冗余存储，方便 SDK 直接读取 |
| `attributeName` | string | 冗余存储属性名称 |
| `attributeField` | string | 上报时的字段 key |
| `description` | string | 当前事件中该属性的描述，可为空 |
| `sourceType` | string \| null | 仅 `custom` 类型有值 |
| `sourceValue` | string \| null | 仅 `custom` 类型有值 |
| `interfaceId` | number \| null | 仅 `sourceType=api_data` 时有值 |
| `interfacePath` | string \| null | 仅 `sourceType=api_data` 时有值，由后端填充 |
| `defaultValue` | string \| null | 仅 `custom` 类型有值 |

### 六、接口定义

#### 6.1 统一说明

- 基础路径：`/api/event-manage`
- 统一返回结构：`{ "code": 200, "message": "success", "data": {} }`

#### 6.2 权限要求

| 接口 | 权限标识 | 说明 |
|---|---|---|
| `GET /api/event-manage/list` | `track-config:view` | - |
| `GET /api/event-manage/all` | `track-config:view` | 供 SDK 使用 |
| `GET /api/event-manage/requirement-options` | `track-config:view` | 获取可关联需求下拉 |
| `POST /api/event-manage/add` | `track-config:add` | - |
| `POST /api/event-manage/update` | `track-config:edit` | - |
| `POST /api/event-manage/delete` | `track-config:delete` | - |

#### 6.3 事件列表

- 方法：`GET`
- 路径：`/api/event-manage/list`

**请求参数**

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `keyword` | string | 否 | 按 `eventName` 或 `eventCode` 模糊搜索 |
| `eventType` | string | 否 | `page_view` / `click` |
| `pageNum` | number | 否 | 页码，默认 1 |
| `pageSize` | number | 否 | 每页条数，默认 10 |

**响应 `data`**

分页结构，内容为 `EventConfig[]`（见 6.8）。

---

#### 6.4 全量事件列表（SDK 使用）

- 方法：`GET`
- 路径：`/api/event-manage/all`

**请求参数**：无

**响应 `data`**

`EventConfig[]`，仅返回 `status=1`（启用）的事件，包含完整 `params` 字段，供 SDK 拉取配置使用。

---

#### 6.5 可关联需求下拉

- 方法：`GET`
- 路径：`/api/event-manage/requirement-options`

**请求参数**

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `keyword` | string | 否 | 按需求标题模糊搜索 |

**响应 `data`**

```json
[
  {
    "requirementId": "uuid-string",
    "title": "需求标题",
    "status": "SCHEDULING",
    "statusLabel": "排期中"
  }
]
```

说明：仅返回状态为 `SCHEDULING`（排期中）或 `DEVELOPING`（开发中）的需求。

---

#### 6.6 新增事件

- 方法：`POST`
- 路径：`/api/event-manage/add`

**请求体**

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `eventName` | string | 是 | 事件名称 |
| `eventCode` | string | 是 | 事件编码 |
| `eventType` | string | 是 | `page_view` / `click` |
| `status` | number | 是 | `1=启用`，`0=禁用` |
| `requirementId` | string | 是 | 关联需求 ID |
| `description` | string | 否 | 事件描述 |
| `params` | string | 否 | 关联属性 JSON 数组序列化字符串（见 5.2），无关联属性时传空字符串或 `null` |

**响应 `data`**

新增后的 `EventConfig` 对象。

---

#### 6.7 更新事件

- 方法：`POST`
- 路径：`/api/event-manage/update`

**请求体**

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | number | 是 | 事件主键 ID |
| `eventName` | string | 是 | - |
| `eventCode` | string | 是 | - |
| `eventType` | string | 是 | - |
| `status` | number | 是 | - |
| `requirementId` | string | 是 | - |
| `description` | string | 否 | - |
| `params` | string | 否 | - |

**响应 `data`**

更新后的 `EventConfig` 对象。

---

#### 6.8 删除事件

- 方法：`POST`
- 路径：`/api/event-manage/delete`

**请求体**

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | number | 是 | 事件主键 ID |

**响应 `data`**

`null`

---

#### 6.9 EventConfig 数据结构

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | number | 主键 |
| `eventName` | string | 事件名称 |
| `eventCode` | string | 事件编码 |
| `eventType` | string | `page_view` / `click` |
| `status` | number | `1=启用`，`0=禁用` |
| `requirementId` | string | 关联需求 ID |
| `requirementTitle` | string | 关联需求标题（查询时关联返回） |
| `requirementStatus` | string | 关联需求状态（查询时关联返回） |
| `description` | string | 事件描述 |
| `params` | string | 关联属性 JSON 数组序列化字符串 |
| `createTime` | datetime | 创建时间 |
| `updateTime` | datetime | 更新时间 |

### 七、数据迁移说明

改造上线前，需将现有 `track_config.params` 中的原参数配置迁移至新属性体系：

1. 遍历所有 `track_config` 记录，解析 `params` 字段中的参数数组
2. 每条参数配置（原 `paramName + sourceType + sourceValue + ...`）作为一条个性化属性（`attributeType=custom`）写入 `track_attribute` 表，`attributeName` 取原 `paramName`，`attributeField` 取原 `paramName`
3. 用新的属性引用结构（见 5.2）替换 `track_config.params` 中的原参数数组内容
4. 迁移脚本与测试用例输出至 `track-backend/docs/增量相关文档及脚本/`

### 八、补充说明

- 事件编码（`eventCode`）在生效数据内唯一，新增和编辑时后端需校验。
- 所有时间字段使用服务器时间。
- 关联需求的状态发生变更（如从"开发中"变为"测试中"）后，已关联的事件不受影响，仍保持关联关系，后续仅在创建新事件时限制可选需求的状态范围。
