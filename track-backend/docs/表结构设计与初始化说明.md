# Track Backend MySQL 表结构设计与初始化说明

## 1. 目标

- 将 `track-backend` 从 H2 内嵌数据库切换为 MySQL。
- 不迁移历史 H2 数据，采用全量初始化。
- 初始化内容包含：管理员账号、角色、菜单按钮、角色权限关系。
- 移除“重置数据”功能相关菜单与接口。

## 2. 数据库与字符集建议

- MySQL: 8.0+
- Charset: `utf8mb4`
- Collation: `utf8mb4_unicode_ci`
- 时区: `Asia/Shanghai`

## 3. 表清单

- `sys_user` 用户表
- `sys_role` 角色表
- `sys_menu` 菜单/按钮权限表
- `sys_user_role` 用户-角色关联表
- `sys_user_data_dept` 用户-数据授权部门关联表
- `sys_role_menu` 角色-菜单关联表
- `track_config` 埋点配置表
- `track_attribute` 属性管理表
- `track_data` 埋点上报数据表
- `track_requirement` 需求管理主表
- `track_log` 通用操作日志表（需求管理使用 `log_type=requirement_manage`）
- `api_interface` 接口来源表
- `dict_param` 参数主表
- `dict_param_item` 参数项表
- `dict_id_sequence` 业务ID序列表（支持 `DICT`、`ATTR`）

## 4. 关键设计点

- 主键策略：
  - `sys_user` / `sys_role` / `sys_menu`：`BIGINT` + 应用侧雪花ID生成（非数据库自增）。
  - 其余业务表保持 `BIGINT AUTO_INCREMENT`。
  - 雪花ID采用 JS-safe 53 位方案，前端可安全传输与展示。
- 内置超级管理员：
  - `sys_user.is_builtin_super_admin`：`1=内置超级管理员账号`，该账号不参与普通用户管理维护。
  - 内置超级管理员拥有全权限与全数据范围，不依赖角色菜单配置。
- RBAC 约束：
  - `sys_user_role` 增加唯一键 `(user_id, role_id)`。
  - `sys_role_menu` 增加唯一键 `(role_id, menu_id)`。
- RBAC 外键：
  - 关联表对主表采用 `ON DELETE CASCADE`。
- 组织与数据权限：
  - `sys_user.primary_dept_id` 表示用户唯一所属部门（来自 `dict_param_item`，`param_id=SYS_DEPT`）。
  - `sys_user_data_dept` 支持用户多部门数据授权。
- 业务数据表 `track_config`/`api_interface`/`track_data` 增加 `dept_id` 用于数据范围过滤。
- 事件管理：
  - `track_config` 新增 `requirement_id`，创建/编辑事件必须关联需求。
  - `params` 结构由内联参数升级为“属性引用列表”，引用 `track_attribute.attribute_id`。
- 属性管理：
  - 新增 `track_attribute` 统一维护上报属性（`user/system/custom`）。
  - 属性业务ID格式：`ATTR + yyyyMMdd + 8位序号`。
- 菜单结构：
  - `menu_type`: `1=目录`, `2=页面`, `3=按钮`。
  - 按钮权限标识通过 `perms` 字段维护，如 `system-role:edit`。
- 需求管理：
  - `track_requirement` 记录需求完整生命周期核心字段。
  - 状态流转由后端服务强校验，前端根据后端返回 `availableActions` 渲染操作按钮。
  - `track_log` 承载操作日志，需求管理统一写入 `log_type=requirement_manage`。
- 参数维护：
  - 主表软删除字段：`status`，`0=生效`，`1=已删除`。
  - 主表系统标记：`is_system`，`1=系统内置参数`（如部门参数 `SYS_DEPT`）。
  - 参数名称唯一性仅对 `status=0` 生效，通过 `active_name` 生成列唯一索引保证。
  - 参数业务ID格式：`DICT + yyyyMMdd + 8位序号`，序号由 `dict_id_sequence` 并发安全生成。
  - `dict_id_sequence.biz_date` 扩展为 `VARCHAR(32)`，用于承载不同业务序列键（如 `ATTRyyyyMMdd`）。
  - 参数维护接口采用后端角色硬校验：仅 `developer` 角色可访问；内置超级管理员账号通过超级管理员身份放行。
- 重置数据能力：
  - 已从初始化菜单与权限中移除 `system-reset-data*`。

## 5. 初始化数据策略

### 5.1 账号与角色

- 默认账号：`admin / 123456`
- 默认内置超级管理员：`admin`（`is_builtin_super_admin=1`）
- 角色：`admin`、`business`、`developer`
- 默认部门参数：`SYS_DEPT`
- 默认部门：`DEFAULT / 默认部门`

### 5.2 菜单与按钮（36 条）

包含以下业务域：

- 仪表盘
- 埋点管理
  - 事件管理（查看/新增/编辑/删除）
  - 属性管理（查看/新增/编辑/删除）
  - 接口来源管理（查看/新增/编辑/删除）
  - 数据回检（查看）
  - 需求管理（查看/新增/状态变更/重新提交）
- 系统管理
  - 用户管理（查看/新增/编辑/删除）
  - 角色管理（查看/新增/编辑/删除）
  - 参数维护（页面菜单，权限由后端角色硬校验）

### 5.3 权限初始化

- 内置超级管理员账号：全菜单全权限（后端硬校验，不依赖角色关联）。
- `admin` 角色：普通系统管理员角色，通过角色菜单配置控制权限。
- `business`：仪表盘 + 事件管理（全）+ 属性管理（全）+ 接口来源（全）+ 数据回检（查看）+ 需求管理（查看/新增/重新提交）。
- `developer`：仪表盘 + 事件管理（查看）+ 属性管理（查看）+ 接口来源（查看）+ 数据回检（查看）+ 需求管理（查看/状态变更）。
- `admin` 用户初始化默认绑定 `admin` 角色（仅作为初始角色分配，不影响超级管理员身份判定）。

## 6. 脚本清单

位于 `src/main/resources/db/mysql`：

1. `01_schema.sql`：建表脚本（DDL）
2. `02_init_data.sql`：基础初始化数据（用户/角色/菜单）
3. `03_init_permissions.sql`：权限初始化（用户角色、角色菜单）

增量变更脚本（docs）：

- `docs/增量相关文档及脚本/20260423_rbac_dept_data_scope.sql`：RBAC + 部门数据权限改造脚本
- `docs/增量相关文档及脚本/20260423_builtin_super_admin_refactor.sql`：内置超级管理员与管理员角色解耦脚本
- `docs/增量相关文档及脚本/20260424_requirement_manage.sql`：需求管理模块表结构/菜单权限增量脚本
- `docs/增量相关文档及脚本/20260426_rbac_snowflake_id.sql`：用户/角色/菜单主键由自增切换为雪花ID策略
- `docs/增量相关文档及脚本/20260427_event_attribute_manage.sql`：事件管理/属性管理改造增量脚本

## 7. 执行顺序

1. 执行 `01_schema.sql`
2. 执行 `02_init_data.sql`
3. 执行 `03_init_permissions.sql`
